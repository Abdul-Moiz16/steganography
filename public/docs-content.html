Stego Explorer — Documentation

Overview

Stego Explorer is a self-contained pipeline browser and experiment launcher for Maastricht University Project 2.2 (Spring 2026). It lets you launch pipeline runs, monitor execution in real time via streaming logs, and explore detection results across all experimental conditions.

The central question is whether the source of the carrier image changes how detectable steganographic embedding is. Three carrier sources are compared: real photographs (COCO/Flickr30k), ML-A (SDXL 1.0), and ML-B (PixArt-α per proposal; FLUX.1-schnell in the prototype — see Proposal Divergences). Five research questions structure the study:

  • RQ1 — Carrier Source (real vs. ML): Does carrier source (real photographs vs. ML-generated images) affect steganographic detectability under matched embedding settings? ML-A and ML-B are pooled into one ML group for this comparison.
  • RQ2 — Generator Effect Within ML: Within ML-generated carriers, does the choice of generator (ML-A vs. ML-B) affect detectability?
  • RQ3 — Payload Interaction: Does payload size change the detectability gap between carrier sources?
  • RQ4 — Embedding Branch: Do the spatial branch (LSB+PNG) and the frequency branch (DCT-LSB+JPEG) show different carrier-source effects? Each branch bundles embedding with its file format.
  • RQ5 — Encryption Invariance: Does AES-256-CBC encryption of the payload affect detectability? Encrypted payloads should look like random bitstreams, so we expect no difference. A positive result here would suggest the detector is reacting to payload structure rather than to embedding distortion itself.

Platform Architecture

The viewer is a single-file Python HTTP server (viewer.py) built on the standard library's http.server.BaseHTTPRequestHandler and socketserver.ThreadingMixIn. No external web framework is required.

Backend (viewer.py)

  • Static serving — Files under /public/ are served with MIME-appropriate cache headers.
  • REST API — GET /api/runs, GET /api/runs/<id>/detail, DELETE /api/runs/<id>, POST /api/pipeline/start, POST /api/pipeline/kill/<job>.
  • Log streaming — GET /api/pipeline/stream/<job_id> streams subprocess stdout as Server-Sent Events.
  • Cross-instance sync — GET /api/events broadcasts filesystem change events to all connected browsers. A background thread polls the runs/ directory mtime every 2 seconds and emits a refresh event on any change.
  • Multi-instance safety — Run IDs embed the server port (e.g. prototype_20260312_p8765). A .running marker file prevents deletion of active runs from any instance.

Frontend (public/)

  • app.js — SPA router, API client, all page renderers, launch drawer, and educational carousel.
  • charts.js — ROC curve and score distribution renderers using the Canvas 2D API.
  • style.css — All styles with dark/light theming via CSS custom properties and html.dark / html.light class toggles stored in localStorage.

Pipeline Stages

The viewer's Launch Run button calls python run.py <profile>. This is the single user-facing entry point. It handles cover acquisition, then delegates to the internal src.pipeline.cli run-all command for the embedding and detection stages. All outputs are self-contained under runs/{profile}_{timestamp}/.

Two run profiles are defined in src/pipeline/profile.py. Conditions = sources(3) × methods × payload levels × encryptions(2).

ProfileGroupsImages/runMethodsFill ratesConditions
prototype2060lsblow (0.25 bpp)6
full_design5001500lsb, dctlow / medium / high36

Cover sources per group: REAL (COCO / Flickr30k photograph), ML-A (SDXL 1.0), ML-B (FLUX.1-schnell in prototype & full_design — PixArt-α per proposal, see Proposal Divergences).

1 · Download real covers

run.py calls src.data.download_real_covers to fetch COCO + Flickr30k images via the HuggingFace Datasets API. Images are written to run_dir/covers/real/ and indexed in run_dir/manifests/covers_real.csv. Idempotent: skipped if the manifest already has enough rows.

# prototype: 12 COCO + 8 Flickr30k
# full_design: 300 COCO + 200 Flickr30k

2 · Generate ML covers

Calls src.data.generate_ml_covers to produce SDXL 1.0 (ML-A) and FLUX.1-schnell (ML-B) images from the real-image captions. Engine is selectable: inference_api (HuggingFace Inference API, default), diffusers (local GPU), or stub (synthetic, no GPU).

python run.py prototype --ml-engine inference_api
python run.py prototype --ml-engine stub

3 · Merge covers manifest

Merges covers_real.csv, covers_ml_a.csv, and covers_ml_b.csv into a single run_dir/manifests/covers.csv with standardized grayscale 512×512 PNG (spatial) and JPEG Q=95 (frequency) variants for each image.

4 · run-all (via src.pipeline.cli)

With the covers manifest ready, run.py invokes src.pipeline.cli run-all which runs four sub-stages in sequence:

  1. build-payload-manifest — generates pseudo-random payload bytes (seed=42) for each fill rate and encryption variant. The Cartesian product of groups × payload levels × encryptions is one row per payload artifact.
  2. build-stego-manifest — cross-joins covers × methods × payload levels × encryptions into one row per embedding job (the full experimental design table).
  3. run-embedding-stage — for each stego manifest row: encrypts the payload if required (AES-256-CBC), embeds it using the specified method, and writes the stego image to run_dir/stego/.
  4. run-detectors — scores every active detector against every stego and cover image. Writes run_dir/predictions/predictions.csv. --skip-unimplemented silently skips detectors that raise NotImplementedError.

5 · compute-metrics & plot-metrics

compute-metrics aggregates predictions into four CSV tables under run_dir/metrics/: per-detector (ROC-AUC, EER, accuracy at Youden's J), per-condition, per-source, and quality metrics. These drive the Results and Conditions tabs in the viewer.

plot-metrics (optional, --generate-figures) generates AUC summary figures: AUC by source/detector and AUC by method/detector, written to run_dir/figures/.

Embedding Methods

LSB Sequential

IMPLEMENTED

The primary embedding method. Converts the cover image to 8-bit grayscale, flattens to row-major order, and overwrites the LSB(s) of the first floor(N × fill_rate) pixels with payload bits. Sequential order is intentional — it ensures RS analysis, chi-square, and Sample Pairs detectors observe the expected statistical pattern.

embed_lsb(
    cover_image   : Image.Image,
    payload_bytes : bytes,
    fill_rate     : float,   # 0.25 | 0.50 | 0.75
    *,
    bit_depth     : int = 1  # bits per pixel replaced
) -> Image.Image

Embedding for each pixel position i:

mask     = ~((1 << bit_depth) - 1) & 0xFF   # clears target bits
stego[i] = (cover[i] & mask) | payload_bits[i]

LSB Keyed

IMPLEMENTED — OPTIONAL EXTENSION

An extension provided in src/embedding/lsb.py but not used by either run profile. It shuffles embedding positions using a SHA-256-derived PRNG seed, spreading distortion uniformly but breaking the sequential-order assumption of training-free detectors. The main pipeline always uses sequential LSB.

embed_lsb_keyed(
    cover_image   : Image.Image,
    payload_bytes : bytes,
    fill_rate     : float,
    *,
    bit_depth     : int = 1,
    key           : str      # passphrase used as PRNG seed
) -> Image.Image

Key derivation: seed = int(sha256(key.encode("utf-8")).hexdigest(), 16). A random.Random(seed) instance shuffles the full pixel index list; the first usable_pixels positions are used for embedding.

DCT LSB (JSteg-style)

PLANNED

JSteg-style LSB replacement in quantised DCT coefficients of JPEG images. DC coefficients and zero-valued AC coefficients are skipped to avoid introducing new non-zero values. Embedding traverses 8×8 blocks in row-major order, using the first fill_rate fraction of eligible AC coefficients as positions. Modified coefficients are re-entropy-coded without a second quantization pass.

embed_dct_lsb_jpeg(
    cover_jpeg_bytes : bytes,
    payload_bytes    : bytes,
    fill_rate        : float,
    *,
    jpeg_quality     : int = 95
) -> bytes   # JPEG bytes

References: [westfeld1999] [fridrich2003]

Steganalysis Detectors

All detectors return a single float score where higher values indicate stronger evidence of embedding. Cover images produce low scores; stego images ideally produce high scores. ROC-AUC and EER are computed from the score distributions.

RS Analysis

IMPLEMENTED

Pixels are extracted as non-overlapping 2×2 blocks (flattened to 4-element vectors). Smoothness f of each block is the sum of absolute differences between adjacent elements. Two masks are then applied to each block:

  • Positive mask — XOR positions 1 and 2 (0-indexed) with 1, flipping their LSB.
  • Negative mask — at positions 1 and 2: subtract 1 if the value is even, add 1 if odd.

Each block is classified by comparing its post-mask smoothness to f:

  • Regular (R) — smoothness increases after masking
  • Singular (S) — smoothness decreases after masking

In an unmodified image Rm ≈ R−m and Sm ≈ S−m. LSB replacement disrupts this balance. Score:

score = |R_m − R_−m| + |S_m − S_−m|

Computed independently per colour channel; the maximum channel score is returned. [fridrich2001]

Chi-Square Spatial

IMPLEMENTED

For each of the 128 value pairs (2k, 2k+1), the expected count under full LSB replacement is the average of the two observed histogram bins. The chi-square statistic accumulates one term per valid pair (where E > 0) using only the even-valued bin:

E         = (n[2k] + n[2k+1]) / 2
χ² += (n[2k] − E)² / E       # one term per pair
df        = valid pairs − 1
score     = chi2.sf(χ², df)   # survival function: high = evidence of embedding

A cover image has unbalanced pairs (high χ², low survival probability, low score). An LSB-embedded image has balanced pairs (low χ², high survival probability, high score). [westfeld1999]

Sample Pairs Analysis

IMPLEMENTED

Analyses the trace multiset T of adjacent pixel pairs across the image in row-major order. T counts pairs (u, v) where one value is even and the other is odd. Sequential LSB embedding predictably shifts these counts. The embedding rate β is estimated by solving a quadratic derived from the shifted multiset statistics.

sample_pairs_score(
    image : Image.Image
) -> float

[dumitrescu2003]

Chi-Square DCT

NOT YET IMPLEMENTED

Applies the same chi-square pairs-of-values test as the spatial variant, but to quantised DCT coefficients of a JPEG image. DC coefficients are excluded; only non-zero AC values are tested. Targeted at stego images produced by DCT-LSB embedding.

chi_square_dct_score(
    jpeg_bytes : bytes
) -> float

[westfeld1999]

Calibration Chi-Square

NOT YET IMPLEMENTED

Improves on the DCT chi-square by constructing a calibration reference: the suspect JPEG is cropped by a non-block-aligned offset (typically 4 pixels), re-compressed at the same quality factor, and its DCT histogram is compared against the original. This subtracts the natural DCT statistics shared by both, leaving only embedding artefacts.

calibration_chi_square_score(
    jpeg_bytes   : bytes,
    *,
    jpeg_quality : int = 95
) -> float

[fridrich2003]

Encryption

Before embedding, the payload can optionally be encrypted with AES-256-CBC. This is controlled by the encryption column of the stego manifest — each job specifies plain or aes256cbc.

encrypt_payload_aes_256_cbc(
    payload : bytes,
    key     : bytes,   # exactly 32 bytes
    iv      : bytes    # exactly 16 bytes (CBC IV)
) -> bytes
  • AES-256 in CBC mode with PKCS#7 padding (128-bit blocks), using the cryptography library.
  • IV is derived deterministically: SHA-256(f"{group_id}:{payload_level}".encode())[:16]. This makes ciphertext fully reproducible for a fixed group and payload level without storing the IV separately.
  • The payload itself is a pseudo-random bitstream generated by the pipeline (seeded by payload_seed=42). The plain variant embeds this bitstream directly; the encrypted variant embeds its AES-256-CBC ciphertext.
  • Encryption is an in-memory operation; PipelineRunner handles all file I/O.
  • Since both the plaintext payload and the ciphertext are statistically close to uniform random bytes, the detectability difference between the two encryption conditions measures only the overhead introduced by PKCS#7 block alignment (RQ5).
